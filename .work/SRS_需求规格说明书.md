## 1. 引言

### 1.1 文档目的
本文档详细描述 SwallowNote Markdown 编辑器的功能需求、性能需求、接口需求和约束条件，作为系统设计、开发、测试和验收的依据。

### 1.2 产品范围
SwallowNote 是一款跨平台桌面应用，提供高性能的 Markdown 编辑体验，集成 Git 版本控制、多源同步、AI 智能辅助等功能。

### 1.3 目标读者
- 产品经理：理解产品功能和优先级
- 系统架构师：进行技术选型和架构设计
- 开发工程师：实现具体功能模块
- 测试工程师：编写测试用例和验收标准
- UI/UX 设计师：设计界面和交互流程

### 1.4 术语定义

| 术语 | 定义 |
|------|------|
| 工作区 (Workspace) | 包含一个或多个根目录的项目集合，有独立的配置文件 |
| 虚拟滚动 (Virtual Scrolling) | 仅渲染可视区域 DOM 元素的技术，用于优化大量数据的展示性能 |
| Token | AI 模型处理文本的基本单位，1 Token ≈ 0.75 英文单词或 1.5 中文字符 |
| 增量同步 | 仅传输变更部分的同步方式，相比全量同步节省带宽和时间 |
| CDN | Content Delivery Network，内容分发网络，用于加速静态资源访问 |

## 2. 总体描述

### 2.1 产品视角
SwallowNote 是独立的桌面应用程序，不依赖其他软件，但可与以下外部系统集成：
- Git 版本控制系统
- WebDAV/S3/FTP 云存储服务
- AI 模型服务（本地或云端）

```

│          SwallowNote 核心功能            │

├──────────┼──────────┼────────┼──────────┤
│ • 极速加载│ • 初始化  │ • Git  │ • 对话   │

└──────────┴──────────┴────────┴──────────┘
```

### 2.3 用户特征

| 用户类型 | 特征 | 使用场景 |
|---------|------|---------|
| 开发者 | 熟悉 Git，需要代码片段管理 | 技术文档、API 文档、README |
| 技术写作者 | 注重排版和多媒体 | 博客文章、教程、报告 |
| 笔记爱好者 | 需要知识管理和同步 | 学习笔记、会议纪要、日记 |

### 2.4 约束条件
- 必须支持 Windows/macOS/Linux 三大平台
- 本地优先原则，用户数据默认存储在本地
- 遵循开源协议（待定）
- 符合 GDPR 等隐私保护法规

### 2.5 假设与依赖
- 用户已安装 Git（如未安装，部分功能降级）
- 网络连接可用时使用云同步和 AI 功能
- 用户具备基本的 Markdown 语法知识

## 3. 详细功能需求

### 3.1 核心编辑功能

#### FR-001: Markdown 实时预览

**功能描述**  
用户在编辑 Markdown 源码时，右侧或分屏实时显示渲染后的效果。

**用户场景**  
1. 用户打开 `.md` 文件
2. 在左侧编辑器输入 `# Hello World`
3. 右侧预览区立即显示一级标题 "Hello World"

**输入**  
- Markdown 源文本

**输出**  
- 渲染后的 HTML 页面

**业务规则**  
- R1: 支持 CommonMark 标准语法
- R2: 支持 GFM (GitHub Flavored Markdown) 扩展
- R3: 支持表格、任务列表、脚注、定义列表
- R4: 支持数学公式（KaTeX 渲染）
- R5: 支持 Mermaid 图表（流程图、时序图、甘特图等）
- R6: 支持代码块语法高亮（至少 20 种常用语言）
- R7: 预览延迟 < 200ms（防抖处理）

**异常处理**  
- E1: 语法错误时显示警告但不阻止渲染
- E2: 不支持的语法保持原文显示

**验收标准**  
- [ ] 输入 Markdown 后 200ms 内更新预览
- [ ] 所有支持的语法正确渲染
- [ ] 代码块高亮准确

**输出**  
- 界面布局变化

**业务规则**  
- R1: 默认模式为"仅源码"
- R2: 分屏比例可拖动调整（30%-70%）
- R3: 记住用户上次使用的模式
- R4: 预览区滚动与编辑区同步（可选开关）

**验收标准**  
- [ ] 三种模式切换流畅无闪烁
- [ ] 分屏比例调整平滑
- [ ] 模式偏好被持久化保存

#### FR-004: 大文件编辑优化

**功能描述**  
支持流畅编辑 10MB 以上的 Markdown 文件，通过虚拟滚动和懒加载技术保证性能。

**用户场景**  
1. 用户打开 15MB 的 Markdown 文件
2. 文件在 3 秒内加载完成
3. 滚动浏览和编辑操作流畅无卡顿

**输入**  
- 大型 Markdown 文件路径

**输出**  
- 文件内容加载到编辑器

**业务规则**  
- R1: 采用虚拟滚动，仅渲染可视区域行（约 50 行）
- R2: 预览区采用分页渲染，每次渲染 1000 行
- R3: 后台线程解析 Markdown，避免阻塞 UI
- R4: 内存占用控制在 200MB 以内（10MB 文件）
- R5: 支持断点续载，中断后可恢复

**性能指标**  
- P1: 10MB 文件打开时间 < 3s
- P2: 滚动帧率 ≥ 50 FPS
- P3: 输入延迟 < 50ms

**异常处理**  
- E1: 文件超过 50MB 时提示用户确认
- E2: 内存不足时提示关闭其他标签页

**验收标准**  
- [ ] 10MB 文件打开时间 ≤ 3s
- [ ] 滚动流畅无明显卡顿
- [ ] 内存占用符合预期

### 3.2 文件管理功能

#### FR-005: 文件树极速加载

**功能描述**  
采用异步加载和虚拟列表技术，实现千级文件目录的秒级加载和流畅操作。

**用户场景**  
1. 用户打开包含 2000 个文件的目录
2. 文件树在 500ms 内显示首屏内容
3. 展开子目录时异步加载，无阻塞

**输入**  
- 根目录路径

**输出**  
- 树形结构展示文件和文件夹

**业务规则**  
- R1: 首屏仅加载根目录和第一层子目录
- R2: 子目录展开时按需加载（Lazy Load）
- R3: 使用虚拟列表，DOM 节点数 ≤ 100
- R4: 文件系统变化监听（chokidar），自动刷新
- R5: 缓存目录结构，减少重复扫描
- R6: 文件图标根据扩展名区分（.md, .jpg, .png, 其他）
- R7: 显示文件大小和修改时间（可选）

**性能指标**  
- P1: 1000 个文件首屏加载 < 500ms
- P2: 展开子目录响应时间 < 100ms
- P3: 文件树内存占用 < 50MB

**异常处理**  
- E1: 权限不足时显示锁定图标
- E2: 符号链接循环引用时截断并警告
- E3: 文件删除后自动从树中移除

**验收标准**  
- [ ] 2000 文件目录加载时间 ≤ 500ms
- [ ] 展开/折叠操作流畅
- [ ] 文件系统变化实时反映

- [ ] 路径复制准确

支持同时打开多个文件标签，提供标签切换、排序、关闭等功能。

- 标签页状态变化

**业务规则**

### 3.3 Git 集成功能

#### FR-008: Git 仓库检测与初始化

**功能描述**  
打开目录时自动检测是否为 Git 仓库，非 Git 目录提供一键初始化功能。

**用户场景**  
**场景 A：已有 Git 仓库**  
1. 用户打开已初始化的 Git 目录
2. 状态栏显示当前分支名称（如 `main`）
3. 文件树显示 Git 状态标识

**场景 B：非 Git 目录**  
1. 用户打开普通目录
2. 侧边栏 Git 面板显示"此目录未初始化 Git"
3. 用户点击"初始化 Git"按钮
4. 目录初始化为 Git 仓库，显示 `main` 分支

**输入**  
- 目录路径
- 用户点击初始化按钮

**输出**  
- Git 仓库状态
- 分支信息

**业务规则**  
- R1: 检测 `.git` 目录是否存在
- R2: 初始化命令：`git init`
- R3: 初始化后创建默认 `.gitignore`（包含常见忽略规则）
- R4: 设置默认分支名为 `main`
- R5: 显示 Git 命令执行结果（成功/失败）

**异常处理**  
- E1: Git 未安装时提示用户安装并提供下载链接
- E2: 初始化失败时显示错误信息（权限不足等）

**验收标准**  
- [ ] Git 检测准确
- [ ] 初始化成功率高（>99%）
- [ ] 错误提示友好

#### FR-009: Git 自动提交配置

**功能描述**  
用户可配置 Git 自动提交策略，包括触发条件和提交信息格式。

**用户场景**  
1. 用户进入设置 > Git
2. 启用"保存时自动提交"
3. 用户编辑文件并按 `Ctrl+S` 保存
4. 系统自动执行 `git add . && git commit -m "auto commit: 2026-05-07 14:30:25"`

**输入**  
- 自动提交模式选择
- 提交间隔配置（定时模式）

**输出**  
- 自动执行 Git 提交

**业务规则**  
- R1: 三种模式：
  - **手动**：用户手动提交（默认）
  - **保存时**：每次保存文件后自动提交
  - **定时**：每隔 N 分钟自动提交（N ∈ {1, 5, 10, 30}）
- R2: 提交信息格式：`auto commit: YYYY-MM-DD HH:mm:ss`
- R3: 仅提交已跟踪文件的变更
- R4: 提交前自动 `git add .`
- R5: 提交失败时保留更改，不丢失数据
- R6: 记录自动提交日志（时间、文件列表、结果）

**异常处理**  
- E1: 无变更时跳过提交
- E2: 提交冲突时提示用户手动解决
- E3: 网络不可用时（远程推送）仅本地提交

**验收标准**  
- [ ] 三种模式切换正常
- [ ] 提交信息格式正确
- [ ] 提交失败不影响文件保存

#### FR-010: Git 差异对比视图

**功能描述**  
显示文件当前内容与 Git 仓库版本的差异，支持并排和行内两种对比模式。

2. 点击 Git 面板中的文件
3. 右侧显示差异对比视图
4. 绿色行表示新增，红色行表示删除
5. 用户点击"暂存"按钮暂存该文件

**输入**  
- 文件路径
- 对比模式选择（并排/行内）

**输出**  
- 差异对比界面

**业务规则**  
- R1: 调用 `git diff` 获取差异
- R2: 并排模式：左侧原版，右侧修改版
- R3: 行内模式：在同一视图中用颜色标记差异
- R4: 颜色约定：
  - 绿色背景：新增行
  - 红色背景：删除行
  - 黄色背景：修改行
- R5: 支持跳转到下一个/上一个差异（快捷键 `F7`/`Shift+F7`）
- R6: 显示变更统计（+N 行，-M 行）
- R7: 支持逐行暂存（Stage Hunk）

**性能指标**  
- P1: 差异计算时间 < 500ms（1000 行文件）

**验收标准**  
- [ ] 差异显示准确
- [ ] 两种模式切换流畅
- [ ] 跳转功能正常

### 3.4 搜索与过滤功能

#### FR-011: 快速文件搜索（Ctrl+P）

**功能描述**  
通过快捷键唤起命令面板，模糊匹配文件名，快速打开文件。

**用户场景**  
1. 用户按下 `Ctrl+P`
2. 顶部出现搜索框
3. 用户输入 `read`
4. 列表显示所有包含 "read" 的文件（如 `readme.md`, `reading-notes.md`）
5. 用户按上下键选择，Enter 打开

**输入**  
- 搜索关键词
- 键盘导航（↑↓ Enter Esc）

**输出**  
- 匹配的文件列表
- 打开选中文件

**业务规则**  
- R1: 模糊匹配算法（fuzzy search）
- R2: 匹配权重：文件名 > 路径名
- R3: 结果显示：文件名 + 相对路径
- R4: 最多显示 50 条结果
- R5: 支持最近打开文件快捷访问（前缀 `@recent`）
- R6: 支持符号搜索（前缀 `@symbol`，搜索函数/标题）
- R7: 搜索结果按相关性排序

**性能指标**  
- P1: 搜索响应时间 < 100ms
- P2: 索引 5000 个文件时间 < 2s

**验收标准**  
- [ ] 搜索准确率 > 95%
- [ ] 响应速度快
- [ ] 键盘操作流畅

#### FR-012: 全局内容搜索（Ctrl+Shift+F）

**功能描述**  
在所有文件中搜索文本内容，支持正则表达式和多种选项。

**用户场景**  
1. 用户按下 `Ctrl+Shift+F`
2. 左侧面板展开搜索界面
3. 用户输入搜索词 `function`
4. 勾选"区分大小写"和"正则表达式"
5. 结果显示所有包含 "function" 的行，按文件分组

**输入**  
- 搜索关键词
- 搜索选项（大小写、全字匹配、正则）
- 文件过滤（包含/排除 glob 模式）

**输出**  
- 匹配结果列表（文件 + 行号 + 预览）

**业务规则**  
- R1: 支持正则表达式（JavaScript RegExp 语法）
- R2: 选项：
  - 区分大小写（Case Sensitive）
  - 全字匹配（Match Whole Word）
  - 正则表达式（Use Regex）
- R3: 文件过滤：
  - 包含文件：`*.md`
  - 排除文件：`node_modules/**, .git/**`
- R4: 结果显示：
  - 文件名（点击打开文件）
  - 行号（点击跳转到该行）
  - 匹配行预览（前后各 20 字符）
  - 匹配关键词高亮
- R5: 替换功能（可选）：
  - 输入替换文本
  - 逐个替换或全部替换
  - 替换前预览

**性能指标**  
- P1: 搜索 1000 个文件时间 < 3s

#### FR-024: AI 智能补全

类似 GitHub Copilot，根据上下文预测并补全下一句或下一段。

**用户场景**  
1. 用户输入："Markdown 是一种"
2. 等待 1 秒
3. 灰色幽灵文本显示："轻量级标记语言，由 John Gruber 于 2004 年创建。"
4. 用户按 Tab 接受，或 ESC 拒绝

**输入**  
- 当前编辑内容
- 用户按键（Tab/ESC）

**输出**  
- 补全建议
- 文本插入或忽略

**业务规则**  
- R1: 触发条件：
  - 停止输入 1 秒后
  - 行尾或段落末尾
  - 最多显示 1 个建议
- R2: 补全类型：
  - 句子补全
  - 段落补全
  - 代码块补全
- R3: 显示样式：
  - 灰色半透明文本
  - 光标后显示
  - 不占用实际内容
- R4: 接受/拒绝：
  - Tab：接受补全
  - ESC：拒绝补全
  - 继续输入：自动拒绝
- R5: 可配置：
  - 启用/禁用
  - 触发延迟（0.5s - 3s）
  - 最大补全长度

**性能指标**  
- P1: 补全建议生成时间 < 2s
- P2: 不影响输入流畅度

**验收标准**  
- [ ] 补全建议合理
- [ ] 接受/拒绝操作流畅
- [ ] 可配置性强

## 6. 安全需求

### 6.1 数据加密
- API Key、密码等敏感信息使用 AES-256-CBC 加密
- 加密密钥派生自用户主密码（PBKDF2）
- 不在日志、错误信息中明文显示敏感数据

### 6.2 网络安全
- 所有 HTTPS 连接强制 TLS 1.2+
- 证书验证严格，不接受自签名证书（除非用户明确信任）
- WebDAV/S3 传输加密

### 6.3 权限控制
- 文件操作遵循操作系统权限
- 不请求不必要的系统权限
- 沙箱环境运行（Electron contextIsolation）

### 6.4 隐私保护
- 本地优先，用户数据默认不上传
- AI 请求明确告知用户数据会发送到第三方
- 提供"隐私模式"，禁用所有网络功能
- 符合 GDPR，提供数据导出和删除功能

## 7. 可靠性需求

### 7.1 数据完整性
- 自动保存间隔：30 秒（可配置）
- 崩溃恢复：重启后恢复未保存内容
- 文件写入原子操作（先写临时文件，再替换）
- 版本快照：每次保存创建备份（保留最近 10 个）

### 7.2 容错能力
- 同步失败不影响本地编辑
- AI 服务不可用时降级为离线模式
- 网络中断时队列化请求，恢复后重试
-  gracefully degrade（优雅降级）

### 7.3 可用性指标
- 崩溃率：< 0.1%
- 数据丢失率：0%
- MTBF（平均故障间隔）：> 720 小时
