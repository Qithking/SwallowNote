## 2026-05-22: 修复 Windows 平台三个问题

**问题描述：**
1. 编辑器工具栏路径显示错误
2. 应用桌面快捷方式图标模糊，开发菜单图标正常
3. 打开文件夹模式时，文件树根目录只显示文件夹名称，不显示完整路径

**根本原因：**

1. **编辑器工具栏路径显示错误**：`EditorToolbar` 的 `getRelativePath` 函数仅处理了 folder 模式下的 `rootPath`，在 workspace 模式下 `rootPath` 为 null，导致直接显示完整绝对路径。`TabBar` 的 `getRelativePath` 同样存在此问题。

2. **桌面快捷方式图标模糊**：之前为 macOS Dock 图标添加内边距时，错误地将 PNG 图标的实际像素尺寸改成了与文件名不一致的值（如 `128x128.png` 实际变成 152x152），违反了 Tauri 对图标文件名的约定。同时 `icon.ico` 没有重新生成，导致里面的图标仍然是旧版本（无内边距），Windows Square 系列图标也未更新。

3. **文件树根目录显示完整路径**：`filetree.ts` 中 `loadRoot` 使用 `rootPath.split('/').pop()` 提取文件夹名称，在 Windows 上路径使用 `\` 分隔符（如 `C:\Users\test\project`），`split('/')` 无法正确分割，导致显示完整路径。此问题存在于前端所有使用 `split('/')` 处理路径的地方，以及 Rust 后端返回路径时未标准化分隔符。

**解决方案：**

1. **编辑器工具栏路径**：修改 `getRelativePath` 函数，在 workspace 模式下遍历 `workspaceFolders` 匹配路径前缀，生成相对路径。
2. **图标修复**：从原始 512x512 源图标重新生成所有图标文件，保持文件名与像素尺寸一致，在图标内部添加 10% 透明内边距。重新生成 `icon.ico`（包含 16/24/32/48/64/256 所有尺寸）和 `icon.icns`。
3. **路径分隔符标准化**：采用双重方案——
   - **后端**：所有 Rust 命令返回路径时统一将 `\` 替换为 `/`，包括 `list_directory`、`create_file`、文件监听器、Git 仓库扫描、下载路径等
   - **前端**：所有 `split('/')` 改为 `split(/[\\/]/)` 兼容两种分隔符；对话框 API 返回路径时使用 `normalizePath()` 标准化

## 2026-05-22: 修复 macOS Dock 图标过大及关闭窗口后 Dock 图标不隐藏问题

**问题描述：**
1. 在 Dock 上图标过大
2. 关闭应用不退出时，在 Dock 上显示的应用图标应该不显示才对

**根本原因：**
1. **Dock 图标过大**：图标文件（icon.png、icon.icns 等）的内容完全铺满到边缘，没有任何内边距（padding）。macOS 应用图标标准应保留约 10% 的内边距，使内容区域约占 80%，否则视觉上会显得图标过大
2. **关闭窗口后 Dock 图标不隐藏**：当 `closeWithoutExit` 开启时，关闭窗口仅调用 `win.hide()` 隐藏窗口，但 macOS 的 `NSApplicationActivationPolicy` 仍为 `Regular`（默认值），该策略会始终在 Dock 上显示应用图标。需要动态切换为 `Accessory` 策略才能从 Dock 隐藏

**解决方案：**
1. **图标添加内边距**：对所有图标文件添加约 10% 的内边距，重新生成 icon.icns
2. **动态切换 ActivationPolicy**：新增 `set_dock_icon_visibility` Tauri 命令，窗口隐藏时设为 `Accessory`（不显示在 Dock），窗口重新显示时设为 `Regular`（显示在 Dock）

**修改内容：**

1. **修改** 图标文件 `src-tauri/icons/`
   - `icon.png`: 512x512 → 614x614（添加 51px 内边距）
   - `32x32.png`: 32x32 → 38x38（添加 3px 内边距）
   - `64x64.png`: 64x64 → 76x76（添加 6px 内边距）
   - `128x128.png`: 128x128 → 152x152（添加 12px 内边距）
   - `128x128@2x.png`: 256x256 → 306x306（添加 25px 内边距）
   - `icon.icns`: 使用 `iconutil` 重新生成，包含所有标准尺寸

2. **修改** `src-tauri/src/plugins/mac_rounded_corners.rs`
   - 新增 `set_dock_icon_visibility(visible: bool)` Tauri 命令
   - macOS 下通过 `NSApplication.setActivationPolicy` 控制：
     - `visible=true` → `NSApplicationActivationPolicyRegular`(0)，在 Dock 显示
     - `visible=false` → `NSApplicationActivationPolicyAccessory`(1)，在 Dock 隐藏

3. **修改** `src-tauri/src/lib.rs`
   - 注册 `set_dock_icon_visibility` 命令
   - 托盘菜单"显示窗口"事件中，调用 `set_dock_icon_visibility(true)` 恢复 Dock 图标
   - 托盘图标左键点击事件中，同样调用 `set_dock_icon_visibility(true)`

4. **修改** `src/lib/tauri.ts`
   - 新增 `setDockIconVisibility(visible: boolean)` API 封装

5. **修改** `src/App.tsx`
   - 窗口隐藏时（`closeWithoutExit` 模式），调用 `setDockIconVisibility(false)` 隐藏 Dock 图标
   - 涉及三处：直接关闭（无脏文件）、保存后关闭、放弃修改后关闭

## 2026-05-22: 修复新版本下载进度闪动及重复下载问题

**问题描述：** 应用状态栏新版本下载进度会闪动，似乎存在重复下载的情况

**根本原因：** 三个层面的问题叠加导致——

1. **事件监听器泄漏**（主因）：`downloadLatestRelease` 函数每次调用都注册新的 `listen` 监听器但从不取消注册。下载失败后重试时，旧监听器仍然存在，新监听器又被注册，导致同一个事件被多个回调处理 → 进度闪动和看起来像重复下载
2. **后端进度事件过于频繁**：`download_latest_release` 命令每个网络 chunk 都 emit 一次 `download-progress` 事件。大文件可能有数千个 chunk，导致前端被频繁更新 → UI 闪动
3. **下载完成后状态未正确设置**：`onComplete` 回调只设置了 `downloadedPath` 和弹窗，但没有设置 `versionStatus` 为 `download-ready`，导致用户取消弹窗后状态仍停留在 `downloading`

**解决方案：** 三层修复——

1. **`downloadLatestRelease` 返回 cleanup 函数**：在下载完成或失败后自动取消注册事件监听器，重试前先清理旧监听器
2. **后端进度事件节流**：只在进度百分比变化 ≥1% 或距上次 emit 超过 200ms 时才发送事件
3. **正确设置下载完成状态**：`onComplete` 回调中设置 `versionStatus` 为 `download-ready`

**修改内容：**

1. **修改** `src/lib/tauri.ts` - `downloadLatestRelease` 函数
   - 返回值从 `void` 改为 `() => void`（cleanup 函数）
   - 保存 `listen` 返回的 unlisten 函数
   - 下载完成（`download-complete`）后自动调用 cleanup
   - 下载失败（`invoke` 抛错）后自动调用 cleanup
   - 组件卸载时可通过 cleanup 函数取消注册

2. **修改** `src-tauri/src/commands/upgrade.rs` - `download_latest_release` 命令
   - 新增 `std::time::Instant` 导入
   - 添加节流逻辑：记录上次 emit 的百分比和时间
   - 仅在 `current_percent > last_emitted_percent` 或距上次 emit 超过 200ms 时才 emit `download-progress` 事件

3. **修改** `src/components/StatusBar.tsx`
   - 新增 `useRef` 导入和 `cancelDownloadRef` 引用
   - 开始下载前先调用上一次的 cleanup 函数（避免重复监听器）
   - 保存当前下载的 cleanup 函数到 ref
   - `onComplete` 回调中添加 `setVersionStatus('download-ready')`
   - 组件卸载时在 useEffect cleanup 中调用 `cancelDownloadRef.current?.()`

## 2026-05-20: 修复 macOS 发布版本红绿灯按钮未隐藏问题

**问题描述：** macOS 发布版本中系统窗口控制按钮（红绿灯：关闭、最小化、缩放）未隐藏，开发调试模式下正常。`tauri.conf.json` 已设置 `decorations: false`，且自定义 TitleBar 组件实现了窗口控制按钮。

**根本原因：** `mac_rounded_corners.rs` 中的 `enable_modern_window_style` 函数使用条件逻辑检查窗口样式掩码：
```rust
let has_decorations = style_mask.contains(NSWindowStyleMask::NSClosableWindowMask);
if has_decorations {
    style_mask |= NSWindowStyleMask::NSClosableWindowMask;
    style_mask |= NSWindowStyleMask::NSMiniaturizableWindowMask;
}
```
在发布版本中，`has_decorations` 被误判为 `true`，导致系统红绿灯按钮样式掩码被重新添加。

**解决方案：** 采用双重保障策略——
1. **强制移除**红绿灯按钮的样式掩码，而非条件性添加：使用 `style_mask &= !(...)` 确保 `NSClosableWindowMask` 和 `NSMiniaturizableWindowMask` 始终被移除
2. **直接隐藏**红绿灯按钮视图：通过 `setHidden: YES` 单独隐藏三个按钮（close/miniaturize/zoom）

**修改内容：**

1. **修改** `src-tauri/src/plugins/mac_rounded_corners.rs`
   - `enable_modern_window_style` 函数：
     - 移除 `has_decorations` 条件判断逻辑
     - 改为强制移除 `NSClosableWindowMask | NSMiniaturizableWindowMask`
     - 添加通过 `standardWindowButton:` 获取按钮实例并 `setHidden:` 隐藏的逻辑
     - 移除未使用的 `config` 变量和 `position_traffic_lights` 调用
   - `enable_rounded_corners` 函数：将 `config.offset_x/offset_y` 引用改为内联 `offset_x.unwrap_or(0.0)/offset_y.unwrap_or(0.0)`

**双重保障机制：**
1. 样式掩码层：强制清除 `NSClosableWindowMask` 和 `NSMiniaturizableWindowMask`
2. 视图层：直接对三个系统按钮调用 `setHidden: YES`
